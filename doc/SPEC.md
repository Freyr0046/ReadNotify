# 產品需求文件 (Product Requirement Document, PRD)

## 1. 專案概述 (Project Overview)
本專案旨在開發一款 Android 平台上的「語音播報 App」，核心功能為透過系統的通知監聽服務（`NotificationListenerService`）與文字轉語音引擎（`TextToSpeech`），在特定情境與安全防禦機制下，將使用者的推播通知自動轉化為語音播報，提升使用者在開車、運動、雙手不便等情境下的資訊獲取效率與安全性。

### 1.1 核心設計原則
* **高防禦性 (Defensive Design)：** 針對 Android 系統不穩定的背景生命週期、各家 OEM 的積極省電策略、以及語音引擎的異常狀態進行極端狀況攔截。
* **低干擾性 (Anti-Spam)：** 避免 App 淪為噪音來源，必須具備嚴格的過濾、中斷與頻次限制機制。
* **隱私保護 (Privacy First)：** 提供完善的隱私遮蔽，確保敏感訊息不會在公共場合非預期地被唸出。

---

## 2. 產品實作階段藍圖 (Phased Roadmap)

本專案採漸進式迭代開發，共分為三個階段，由打通核心系統管道開始，逐步擴展至環境感知與精細化過濾。

### 階段一：Phase 1 - MVP 核心骨幹（當前目標）
* **定位：** 驗證 Android 系統通訊、背景常駐服務穩定度與基礎 TTS 發聲。
* **核心功能：** `NotificationListenerService` 基礎框架、App 白名單過濾、基礎字數截斷、基礎播報模板、內建測試工具。

### 階段二：Phase 2 - 環境與硬體自動化
* **定位：** 解決「吵到人」與「被干擾」的痛點，建立硬體與情境防禦防線。
* **核心功能：** 音訊輸出路由感知（耳機/喇叭）、動態音訊壓低 (Audio Focus Ducking)、螢幕與通話狀態感知、搖一搖/實體鍵物理中斷、本地通知歷史日誌（Debug 命脈）。

### 階段三：Phase 3 - 精細化運營與高級過濾
* **定位：** 全面對齊市場頂配功能，打磨至商業成熟度。
* **核心功能：** 關鍵字黑名單、重複通知抑制 (Anti-Spam 連續轟炸防護)、鎖定畫面隱私模式、排程勿擾/對齊系統 DND、桌面 Widget 與 Quick Settings Tile 快捷開關。

## 3. Phase 1: MVP 詳細需求規格 (Detailed Specifications)

### 3.1 功能模組與業務邏輯

#### 3.1.1 系統通知監聽服務 (Notification Listener Service)
* **功能描述：** 註冊並啟動 Android `NotificationListenerService`，在背景即時攔截系統通知。
* **資料欄位抓取：**
    * 來源套件名稱 (`StatusBarNotification.getPackageName()`)
    * 通知標題 (`Notification.extras.getString(Notification.EXTRA_TITLE)`)
    * 通知內文 (`Notification.extras.getCharSequence(Notification.EXTRA_TEXT)`)
* **極端情況與防禦行為：**
    * **權限遺失：** 若使用者在中途關閉「通知存取權」，App 於前景重啟時必須偵測並強制跳出全螢幕阻擋視窗，引導引導至系統設定頁，不允許進入主介面。
    * **空值處理：** 若抓取之標題或內文為空（如貼圖、圖片訊息、無文字通知），此條通知直接「靜音跳過」，不觸發 TTS。
    * **自我無限循環防護：** 本 App 自身發出的任何通知，強制鎖死在全域黑名單中，絕不攔截、絕不播報，防止「播報測試通知 -> 產生新通知 -> 再次播報」的無限迴圈。

#### 3.1.2 應用程式白名單 (App Whitelist Filter)
* **功能描述：** 提供包名過濾機制。預設為「全域關閉」，使用者必須在 App 內手動勾選欲啟動播報的應用程式（如 Line、WeChat）。
* **資料結構：** 本地維護一份 `Set<String>` 儲存已勾選的 Package Name。
* **極端情況：** 當白名單為空時，攔截服務保持運行，但對所有傳入通知進行「靜音跳過」。

#### 3.1.3 文字轉語音引擎管理 (TextToSpeech Engine Manager)
* **功能描述：** 初始化 Android 原生 `TextToSpeech` 引擎，並負責語音佇列管理。
* **播報佇列策略：** 採用 `TextToSpeech.QUEUE_ADD` 模式。若前一條通知尚未唸完，新傳入的合格通知將排隊依序播報，不得直接中斷或覆蓋前文。
* **極端情況與防禦行為：**
    * **無 TTS 引擎：** 若系統完全未安裝 TTS 引擎，App 初始化時應透過 `AlertDialog` 提示使用者，並提供跳轉至 Google TTS 下載頁面的 Intent。
    * **不支援的語言 (Language Unsupported)：** 傳入之通知文字若為 TTS 引擎不支援或未下載的語系，引擎將回傳錯誤代碼。App 應採取「靜音跳過」防禦，絕對不允許使用不相干語系硬唸出亂碼或語音破音。

#### 3.1.4 播報限制與模板 (Templates & Constraints)
* **基礎模板：** 固定格式為 `"{AppName}傳來新訊息，內容是：{Content}"`。
* **最大字數限制：** 預設限制為 50 字。
* **字串截斷防禦邏輯：**
    * 若 `{Content}` 長度 $\le 50$ 字：完整播報。
    * 若 `{Content}` 長度 $> 50$ 字：截取前 50 個字元，並在結尾處**由程式碼自動拼接固定語音尾碼** `「...等省略內容」`，使使用者明確知曉資訊未完。

#### 3.1.5 內部測試工具 (Self-Diagnostic Tool)
* **功能描述：** 於 App 主介面提供「發送測試通知」按鈕。
* **運行機制：** 按下後，App 將延遲 2 秒發送一條模擬的第三方通知（模擬特定標題與內文）。用以驗證使用者的權限是否開通、TTS 引擎是否能正常在背景發聲。

## 4. UI 狀態機定義 (UI State Machine)

為確保前景介面與背景服務狀態高度同步，定義以下五大 UI 狀態：

1.  **`INIT_CHECKING` (初始化檢查中)**
    * *進入時機：* App 啟動時。
    * *行為：* 檢查系統通知存取權限與 TTS 引擎可用性。
2.  **`PERMISSION_DENIED` (權限缺失阻擋)**
    * *進入時機：* 檢查發現無通知監聽權限。
    * *UI 表現：* 全螢幕覆蓋，阻擋所有操作，僅顯示一個「前往開啟權限」之核心按鈕。
3.  **`IDLE_CONFIG` (就緒配置狀態)**
    * *進入時機：* 權限正常，且目前無任何語音正在播放。
    * *UI 表現：* 顯示 App 白名單清單、字數限制設定區、以及「發送測試通知」按鈕。
4.  **`TTS_PLAYING` (語音播報中)**
    * *進入時機：* 背景觸發播報，TTS 引擎正在輸出語音。
    * *UI 表現：* 介面同步更新，動態顯示「正在播報：來自 XXX 的通知...」之狀態提示。
5.  **`ENGINE_ERROR` (引擎異常狀態)**
    * *進入時機：* TTS 初始化失敗或遺失語音元件。
    * *UI 表現：* 彈出錯誤對話框，提示修復 TTS 引擎。

## 5. 錯誤狀態與異常矩陣 (Exception Matrix)

| 異常事件 (Event) | 潛在風險 (Risk) | 防禦性處置方案 (Defensive Action) |
| :--- | :--- | :--- |
| **使用者中途關閉通知權限** | 服務失效且產生 SecurityException 崩潰 | App 前景生命週期 `onResume` 強制重新 Check。若權限喪失，立即切換至 `PERMISSION_DENIED` 狀態，終止一切設定操作。 |
| **收到連續 20 條群組群發** | 記憶體溢位或 TTS 佇列無限堆積 | Phase 1 暫設上限：佇列中最多積壓 5 條播報，超過 5 條時，後續新傳入通知直接捨棄。 |
| **通知內文包含超長亂碼或網址** | TTS 唸出冗長英文字母（如 h-t-t-p-s...）嚴重干擾 | 透過正則表達式，在進入 TTS 前將內文中的 URL 直接替換為語音字串 `「網址」`。 |
| **TTS 初始化時無網路且本地無語音包** | 引擎回傳錯誤，導致 App 靜音 | 捕獲 `OnInitListener` 的錯誤碼，不崩潰，並提示使用者切換至系統預設支援的本地語系。 |
